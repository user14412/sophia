## v1.0 问题反思

我的视频制作Agent的v1.0版本已经完成了。感觉视频的质量有点低。

首先动态的视频生成不了，但现在最强的模型生成动态视频也不过几十秒，想自动拼接还是有点难。

其次是其他类型的视频做不了，我用Langgraph写死的workflow，只能生成 脚本+配音/字幕+配图 组合的视频。

最后是现在视频的质量有点低：

- 脚本：
  - 脚本由deepseek一次性写就，长度有限，质量压缩难以保证。这是模型上下文的限制，既然如此，要么考虑其他人有没有专业写（长）视频脚本的skill或mcp接入；要么自己把脚本也拆成一个workflow。但拆成workflow有一个缺点，就是失去灵活性，从此只能写一类脚本了，但换来的是能把这类脚本写好。
- 配音：
  - edge-tts配音音色不好听，更换音色需考虑GPT-SoVITS；
  - edge-tts只能生成单人配音，如需多人配音需考虑ChatTTS
- 配图：
  - qwen文生图质量不好把控，而且图片质量直接决定视频70%的观感。即使AI生图也得人工审核挑选才能选出精品；
  - 视频封面和标题也是吸引读者点击进来的关键。标题感觉生成的还不错，视频封面我考虑为合集专门设计一个模板吸引受众点击；

由于动态视频无法生成，所以无法自由切换不同风格的视频。因此后续优化也只能延续这种写死workflow，做统一一种视频的思路。分析市场，明确受众，了解他们的心理、需求，确定要做的视频类型，接着才好优化workflow，专门把这一种视频做好。感觉由于模型能力限制，workflow灵活性和视频质量有矛盾关系。

另外还有一种矛盾关系，就是AI生成的视频质量比专业up主做的质量低。一种解决方式是你再人工微调，另一种解决方式是充分发挥AI量大管饱的特性，绕开竞争赛道，专注于优势领域，提供独特价值。



## 视频类型选择：

### 视频类型：更静态->更动态

- 当前工作流
- Playlist：曲库里挑选几首歌拼一拼，生成Playlist
- 读书：脚本给定，AI生成音频
  - 有声书：（动漫人物）声线读书
    - 生成配图：如v1.0配图方案 --AI读小说
    - 固定配图：质量高一点，读的书没什么情节或者比较长的时候使用。其实无所谓
  - 播客：脚本的依据给定，AI生成（长）脚本，AI生成音频
  - 长播客：NotebookLM 论文/书 直接转播客：“**此在Lab**”
  - 中等长度播客：“睡前听完一本书”
  - 说书电台：瓦剌留学生活 --相声声线 --背景音乐适配
- 电台：
  - 书、散文诗、音乐、画面适配。这个是我想搞的那种艺术品，感觉比较难，总之就是将 书 / 散文诗、音乐 纯音乐 / 摇滚乐、画作，进行一个杂糅，但是因为AI没有想表达的思想、价值、观点，所以我觉得直接生成高质量的这种有点难。但是可以退而求其次，做一种Vibe Reading，此在Lab式播客解读一段、原文摘选体验一段、音乐增强听一段音乐，画面随之变化。Vibe Reading，用视频带你穿越去书里体验一段日子。但是真的感觉很难。
- **解说-杂谈类**
  - 逐项目划分知识库命名空间
  - 基于知识库中给定的资料
  - 生成二人对谈式的长播客
  - 旨在压缩观众了解对应长内容的时间
  - 例如：二次元简史杂谈（规模大二杂，观众有兴趣却不知从何下手）、东方求闻史记、南明弘光史杂谈，《趣味生活简史》、《人体简史》这作者的书里简直是百科全书，都有取之不尽的有趣而规模宏大的题材。《人类简史》等书也可以作为取材提纲。
  - 这类还有一个好处就是可扩展性强，其他的视频类型workflow写死了就不好往其他题材上靠了，这个解说杂谈类可以对任何大二杂的主题生成深度杂谈。
  - 这不NotebookLM吗？怎么和它打差异化？NotebookLM没有提供接口！
- 动态视频：
  - 纯AI生成动态视频：据考察，暂时有难度
  - 已有视频AI增强：
    - AI换脸 / AI变声/  AI映射为特定动漫角色在特定场景下：好处是可以一段一段地根据原视频生成，视频的语义衔接可以保证，视频的观感质量也有原视频下限托底

话说现在世界上有没有能直接自动生成做的还不错的视频的项目？如果有我想借鉴一下，我搜了一下没搜到。

产品形态：重听觉，轻视觉

**视觉降级与模板化：** 别让 `image_node` 根据每一段 SRT 去盲盒抽卡了。你可以设计一套精美的静态 UI 模板（比如一个复古收音机、一个简单的音频波形可视化，或者一张高质量的哲学意境封面）。这不仅能抹平 AI 绘画 70% 的不可控瑕疵，还能大幅节省计算成本和耗时。

脚本质量瓶颈：writer_node扩展

**音乐增强：** 氛围感（Vibe）高度依赖背景音乐。可以在工作流中增加一个 `music_node`，根据文案的情感标签（如“深沉”、“赛博朋克”、“中世纪”），从你预设的本地免版权高质量曲库中随机挑选合适的 BGM 进行混音。

可扩展性 + 框架稳定

你现在不是在做“视频生成器”，而是在做：

> 一个“可编排的知识生成系统”，视频只是输出形式之一

v2.0 你真正应该做的是这件事：

👉 把系统拆成“三层”，而不是“多个节点”

分层可以更进一步地提升模块化，防止牵一发而动全身

关键点在于：

> 这三层之间只通过“数据结构”通信，而不是 prompt 或状态变量

### 代码优化

- `response_format`（结合 Pydantic BaseModel）

- 把调用 `edge-tts`、DashScope 的代码封装成普通的 Python 类，在对应的 Node 里调用即可

### 接口文档

你的代码行数、模块数已经多到一定程度了，输入输出非常复杂，类型也不好记，维护一个好查询的接口文档已经是必须的了
