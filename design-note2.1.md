## 架构拾遗

- [x] 怎么记外部服务，比如gpt_sovits的路径、使用方法、微调方法，这东西复杂的令人发指，不记下来几天忘光。后面考古式开发就是灾难。而且后续引入数据库等更多外部服务之后，我真不知道该怎么记。
  - [x] 专门的docs记录外部服务怎么配置、怎么接入
  - [x] 路径、API不要硬编码在代码里。外部路径写在.env/config里
  - [x] 项目大了以后服务千万别部署在本地，拥抱docker-compose
- [ ] 路由
  - [ ] 啥叫路由，一个阶段有不同的实现方式，怎么确保程序走正确的管线。比如image既可以文生图，也可以静态图，这时候image就不再是一个impl，实现某个具体生图逻辑的节点了，而是主管派发，image_mode=x走impl1节点，=y走impl2节点。
  - [ ] 用模块间路由代替模块内if-else是一种高级的抽象方法。
    - [ ] 不抽象（路由分派），直接写if-else会导致代码长了就非常混乱。
  - [ ] 这时候image节点和抽象类很像，调用时决定具体实现。
- [ ] **Strategy Pattern（策略模式） + 单节点封装**
  - [ ] pipeline 要稳定
  - [ ] 实现要可切换
- [ ] **把控制流（路由逻辑）和业务逻辑（数据处理）死死耦合在了一起**。一旦图的结构发生变化，你的节点代码就要跟着大改，这很不优雅。在 LangGraph 的设计哲学中，解决这个问题有一个核心准则：节点（Node）应该像纯函数（Pure Function），它不关心自己是从哪条边进来的，它只关心当前的“状态（State）”长什么样。
  - [ ] 专设字段：尽量不要用step来决定节点的业务逻辑，不该图结构，可以把是否是反馈的信息存在状态的某个字段里
  - [ ] 拆分节点：如果一个节点因为入边不同，执行的逻辑差异非常大，那说明这根本就不该是一个节点。你需要把这个“胖节点”拆开。比如**把first_write和re_write，拆分成两个节点**，图稍微大点，但调试非常清晰。
  - [ ] 路由节点：如果不同来源需要构造不同的prompt，可以添加一个专职的适配器节点专门负责构造合适的prompt，再路由到合适的节点

## python拾遗

- [ ] `image_mode: Literal["generate", "static"]`：`image_mode` 这个参数只能填 `"generate"` 或者 `"static"` 这两个字符串中的一个。如果你填了其他的字符串（比如 `"dynamic"`），代码编辑器通常会画红线报错，提醒你值不合法。
- [ ] graph 表达的是“阶段（stage）”，而不是“实现（implementation）”
  - [ ] 不注册 impl，至少状态图是干净的，是纯净的接口节点串起来。这样你不会被图里一大堆impl节点搞晕。
