# 设计思路

## 管线设计

我正在实现我的我的西哲播客视频制作Agent，现已实现2.1版本，2.1版本的文档如附件所示。2.1版本已经跑通了基础的RAG功能，write的每个阶段都RAG西方哲学史讲演录，效果提升极其显著。下面我对2.1版本进行复盘，2.1版本的主要瓶颈已经不在RAG上，而是基于plan-outline-write的内容生产管线的固有限制。主要集中在plan、outline节点的压力过大，无法有效控制全文质量。不是writer能力不行，而是内容的结构不行。我3.0版本的改进思路是，专注于播客内容的优化，固定视频内容的格式，从而减小原先plan、outline节点的压力。

核心优化思路：把“长文本生成问题”，转化成“多轮短文本生成问题”

> 每一轮 prompt 都是稳定的
>
> 每一轮都可以单独评估 / debug
>
> 甚至可以插入人工校验 / rerank

首先打散plan-outline节点为三个步骤：视频的内容不再需要一个自由的策划。
1、给定一本书（可以是任何学科的，哲学、心理学、等等，主要是论述类的，可能要讲2-6个小时），固定按章节分，一个章节就是一个“模块”（30分钟-1小时）

2、每个模块大致是围绕一个中心展开的，再次调llm，把模块细分成循序渐进（深入浅出，就要zero-to-hero，把这段提示词告诉AI很关键），得到“粗课题”list，把粗课题讲完就可以很好地论述一个模块的内容

3、把粗课题细化成一个个具体可回答的具体问题list，[{question_to_ask, question_to_ans}]（current task）的格式，生成时仍然要注意问题之间的衔接和递进，保证第一个问题一定是零基础的，最后一个问题一定是带点深度的。Agent问答系统（纳西妲/海森）接收current task，生成他本次发言的脚本。

> 没必要单独搞一个节点或任务类型去做衔接。在生成下一个“粗课题”的第一个 Q&A 时，只要在 Prompt 里额外注入一个指令（例如：“这是新章节的开始，请在回答本问题前，用一句话总结上文并自然过渡”），同一个 Agent 就能顺滑地完成承上启下。减少任务类型，能大幅降低 LangGraph 图结构的路由复杂度。

这样就把writer写作任务结构化地分成了3类

1、整个视频的开头、结尾，做全书导入和全书总结，这个直接用现有管线生成脚本即可。

2、章节内部，每个Agent根据{question_to_ask, question_to_ans}，生成的发言。这是写作的核心。

3、章节之间的衔接，章节粗问题之间的衔接，承上启下，这个让Agent做一个承上启下的任务就行，其实只要实现了写作类型2，这个类型就很简单。章节和粗问题需要衔接，子问题万万不需要衔接，子问题本身就是抛接球式的设计。实现的时候让Agent遍历粗问题List，根据前后Message作为上下文，生成一段短衔接就行。

3.0阶段我为了快速验证思想，每个视频不做一本书，而是只做一个章节，这样起码少一轮章节之间的衔接。

上面这个设计是否可取，有没有引入不必要的复杂性。

## Aegnt上下文设计

下面对writer节点上下文的讨论集中在第2类。

我如果要在LangGraph中实现一个特定的问答Agent，我想给它4种上下文。

### 1、System Prompt

这部分我希望是Aegnt初始化的时候就加载的，不论进行多少论对话都不变，也不能被忽视的准则

**1.1 Agent 人设和性格**：如纳西妲（女）是智慧之神，比喻、好奇心、同情心、感性。艾尔海森（男）是书记官，严谨、理性、注重逻辑、沉稳

**1.2 Agent 基本职能**：根{question_to_ask, question_to_ans}，和对方一起主持一档播客的，一次需要生成“抛接球”的一轮发言中的自己要读的脚本

**1.3 Agent回复模板与框架**：字数限制区间、优秀回复样例、few-shot

### 2、Current Task：当前任务

{question_to_ask, question_to_ans}的具体内容，Agent每次发言围绕的绝对核心，回答本次问题，并在末尾自然过渡到下一个问题，并把问题抛给对方

### 3、Memory：对话历史

> 核心目的就一个，**顺承语境**，用message滑动窗口就足矣，不要引入RAG

单独chunk（回复的语义粒度比书细，chunk粒度也比较小），单独通过向量数据库存储、检索

3.1 上一轮对方说了什么(AIMessage[-1], importance=0.9)

3.2 ~~更早之前的记忆(importance=0.7)~~（这是完全没必要引入的复杂性）

> **滑动窗口**对话历史，不用向量数据库存储
>
> 直接复用 LangGraph 自带的 `messages` 状态流转。
>
> 纠偏与建议：强烈不建议对对话历史使用向量数据库（RAG）！
>
> - 原因：你规划的一个模块（30-60分钟播客），其纯文案大约在 8000 - 15000 Tokens 之间。目前的 DeepSeek 完全可以无压力 hold 住这个上下文窗口。把短期对话切块存入 RAG，不仅会引入极大的检索延迟，而且会导致语义断层（比如模型检索到了 10 分钟前的一句话，却不知道那句话的上下文语境是什么，这在播客对话中是致命的）。
> - 滑动窗口，这完全满足“上一轮对方说了什么”以及“顺承语境”的需求。

### 4、Knowledeg Base 知识库

chunk，向量数据库存储、检索

4.1 本期视频直接参考的章节 0.9

4.2 本书的其他章节 0.8

4.3 tavily搜索的网络资料 0.5

4.4 用户给的其他资料、书籍 0.7

这组上下文很复杂，分别怎么存(store)？怎么取(query)？怎么把上述上下文嵌入llm每轮的prompt，让llm生成当前子问题的script？

先实现一个**独立的问答机器人**，给具体问题就能出结果

- [ ] bugs
  - [ ] RAG只能查出来一条康德
    - [x] add_rag
  - [x] json解析不稳定
    - [ ] topic_plan的json里可能第一个直接没有question_to_ans字段，最后一个直接没有question_to_ask字段，在writer的时候判断首尾，做好**鲁棒性**
  - [ ] 粗课题之间的衔接缺乏上一个粗课题，只能幻觉
    - [ ] 提供上一轮粗课题的zero_to_hero逻辑，防止瞎掰
  - [ ] 没有输出问题list，直接输出的播客脚本
    - [ ] prompt给正例反例
  - [ ] 问题设计的不好，太过于专业，没有遵循从0到1引入的教育原则，上来就深入探讨哲学名词
    - [x] 哲学名词，出来之前必须解释。解释的方式建议一个人问这个哲学名词是什么，另一个人用更通俗易懂的方式解释它。
    - [ ] 探讨应遵循逻辑，循序渐进，切忌上来就上难度
